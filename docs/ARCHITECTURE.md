# 架构说明

## 为什么不是纯 MOD

这个需求同时包含四类能力：

- 在 Minecraft 里发物品、放方块、扫描建筑。
- 在外部聊天工具里对话。
- 让 Codex/AI 分析自然语言或图片。
- 持久化建筑蓝图，并允许后续定位、修改、复刻。

如果全部写进 Minecraft MOD 或插件里，后续接机器人、图片识别、数据库和本地 Codex CLI 都会很别扭。当前项目采用“插件执行 + 外部控制器规划”的模式。

## 组件边界

### Paper 插件

职责：

- 注册 `/bw` 命令。
- 收集玩家、坐标、世界名等游戏上下文。
- 调用 controller 获取动作。
- 执行动作：发物品、放方块、发送消息。
- 轮询 controller 中由外部机器人创建的任务。

不负责：

- 不直接调用 Codex。
- 不保存复杂蓝图数据库。
- 不接多个聊天机器人。
- 不做图片分析。

### Rust controller

职责：

- 暴露 HTTP API。
- 接收 Minecraft 插件和外部机器人消息。
- 管理蓝图和任务队列。
- 调用规划器，把自然语言变成 `GameAction`。
- 后续调用 Codex CLI 或视觉模型生成蓝图。

## 核心数据模型

### 蓝图

蓝图是建筑的持久化表示：

```text
Blueprint
  id
  name
  description
  size
  materials[]
  blocks[]
  tags[]
```

`blocks[]` 使用相对坐标，例如 `{x: 0, y: 0, z: 0}`。真正放置时由插件把相对坐标叠加到任务原点。

### 游戏动作

controller 不直接操作 Minecraft 世界，只返回动作：

```text
give_item
place_blocks
chat
```

插件是唯一真正改 Minecraft 世界的地方。

## 后续实现图片复刻

图片复刻建议分四步：

1. 图片理解：识别建筑外形、层数、屋顶、门窗、材质风格。
2. 材料映射：把图片里的材质映射到 Minecraft 方块 ID。
3. 蓝图生成：输出 `Blueprint` JSON。
4. 游戏执行：插件按蓝图放置方块。

这样图片分析失败时不会直接破坏游戏世界，只会生成一个可检查、可修改的蓝图。

## 后续实现“识别面前建筑”

需要在 Paper 插件增加扫描接口：

1. 玩家执行 `/bw scan` 或自然语言要求修改建筑。
2. 插件按玩家视线和半径扫描方块矩阵。
3. controller 用扫描结果与已保存蓝图做匹配。
4. 识别出最接近的蓝图和放置原点。
5. 对局部区域生成修改动作。
