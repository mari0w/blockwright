Blockwright

Blockwright coordinates Minecraft Java Edition build requests through a Rust controller. It records blueprints and build jobs, uses MCP tools for player and world-state queries, and delegates block placement and verification to Fabric/Paper execution plugins.

Core components

Natural-language intake, world-state queries, blueprint records, and execution verification.

The controller manages request handling, planning context, blueprint storage, and build records. Fabric/Paper plugins read and modify the Minecraft world through server APIs.

Command intake

Minecraft players submit requests through /bw; external chat integrations can use the controller API.

World-state tools

MCP tools expose player state, inventory, nearby blocks, blueprint data, and build records.

Blueprint model

Blueprints store relative coordinates and block states; the player or task origin is applied only during execution.

Execution verification

After placement, the execution plugin reads the world state and succeeds only when the report matches the build record.

Quick start

The project supports Fabric mod and Paper plugin execution paths.

Run the controller locally, then connect it to a Minecraft Java Edition world through Fabric or Paper. Fabric is the direct mod path; Paper is available for standalone servers.

Start the Web UI

Run ./scripts/run-web.sh from the repository root. The default local URL is http://127.0.0.1:8765/web.

Install the Fabric mod

Run make to build and install into the current HMCL game directory, or specify one with make HMCL_DIR=....

Send in-game requests

Use /bw inside the world for builds, items, or edits to existing structures. Settings live in the Web UI.

Architecture

The controller stores request context and records; execution plugins read and write world state.

This boundary separates planning, storage, and integration logic from Minecraft server API operations.
