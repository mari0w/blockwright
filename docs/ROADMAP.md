# 路线图

Blockwright 的路线图按“先跑通闭环，再增强规划，再提高可发布质量”推进。当前优先保证 Java 版/Fabric 本地世界的体验稳定，再扩展 Paper 和多聊天工具场景。

## 已完成

- Rust/Axum controller 基础 API。
- Fabric 模组支持 Java 版、单人存档和局域网开放世界。
- Paper 插件保留给独立服务端场景。
- 游戏内 `/bw` 命令。
- 外部机器人通用 HTTP 入口。
- Minecraft 执行端任务轮询。
- `give_item`、`place_blocks`、`run_command` 和 `chat` 动作。
- 蓝图 JSON 保存、相对坐标和材料状态表达。
- 构建记录保存和执行端逐块校验回写。
- Codex CLI 自然语言规划。
- 隔离的 Blockwright Codex home 和项目内 skills。
- 已有建筑改造的附近扫描和构建记录匹配基础链路。

## 近期计划

### 1. 执行端稳定性

- 补齐 Fabric 和 Paper 的行为一致性测试。
- 增强放置失败、被保护区阻挡、世界区块未加载等异常报告。
- 增加更清晰的玩家侧中文错误提示。
- 提供撤销或回滚基础能力。

### 2. 建筑改造能力

- 支持更明确的局部选择：窗户、屋顶、地板、墙面、入口、楼层。
- 支持材质替换、扩建、开门、加窗、加灯、换屋顶等常见改造。
- 对匹配多个构建记录的场景提供追问选项。
- 保存改造前后的构建关系，便于后续继续编辑。

### 3. 图片到蓝图

- 接收图片 URL 或本地附件。
- 识别建筑形状、层数、屋顶、门窗和主要材质。
- 输出候选蓝图，而不是直接修改世界。
- 增加玩家确认后再建造的流程。

### 4. 聊天工具

- 钉钉 Stream 模式继续完善。
- Telegram 优先走 polling/getUpdates。
- Discord 优先走本地 bot 进程。
- 保持所有聊天入口都只调用 controller，不直接接 Minecraft 插件。

### 5. 数据层

- 在 JSON 文件稳定后引入 SQLite。
- 保留当前蓝图和构建记录领域模型。
- 增加迁移脚本和导入导出能力。

## 发布前质量目标

- controller 覆盖率不低于 80%。
- Fabric/Paper 核心解析、配置、命令策略和放置校验都有自动化测试。
- README、安装文档、架构文档和贡献说明完整。
- CI 能覆盖 Rust controller、Fabric 测试和 Paper 测试。
- 持续维护中英文 README 和 Web 端中英切换文案。
- 明确安全报告方式。
